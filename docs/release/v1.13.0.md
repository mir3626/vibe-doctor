# v1.13.0 - Intent-Alignment Briefing + User-Accepted Review Close

Date: 2026-07-23

Two coupled features from a settled user consensus, both guarding the same
failure mode: **silent intent drift**, not bugs. The user owns product intent;
Web Pro designs deep (chronic over-engineering risk); repeated Pro↔CLI rounds
drift technical until the user loses the thread. With finite tokens/time the top
question at every boundary is "is this still what the user intended" — so the
CLI turn now answers it mechanically, and a mechanically non-blocking review no
longer forces a Web roundtrip just to close.

## Feature A — User-accepted review close (`accept-review`)

4th application of the declare-in-event/enforce-in-harness pattern (after
`finalGatePolicy`, `productPlane`, `coordinatedClose`).

- A `cli`-actor `approval` event may carry `reviewAcceptance { authorizedBy:
  'user', acceptedFindingIds, reason }` (EVENT-COMPLETE schema, additive).
  Placement is kind-enforced: approval-only, REQUIRED for cli-actor approvals,
  forbidden for pro/codex actors, mutually exclusive with `coordinatedClose`
  (multi-flow coordination stays Pro-only; `ApprovalEventIdSchema` remains
  pro-only so a cli approval can never authorize coordinated member closes).
- Eligibility is validated fail-closed at EVERY snapshot load
  (`validateReviewAcceptance`, mirroring coordinatedClose): previous event is a
  pro feedback; ZERO P0/P1 findings (regardless of plane/productPlane arming);
  feedback disposition not `design-revision-required`/`blocked`; approval HEAD
  equals the reviewed feedback HEAD (post-review code changes force the standard
  remediation loop); `acceptedFindingIds` exactly equals the feedback finding
  set (full-set deferral, no partial acceptance); disposition `approved` for an
  empty finding set, else `approved-with-deferrals`.
- New subcommand `accept-review [flow]` (dry-run: eligibility, findings table,
  recommendation) and `accept-review [flow] --publish --user-approved`
  (publishes the acceptance approval with APPROVAL.md restating every accepted
  finding as a deferral, then `close --publish` runs unchanged).
  `--user-approved` records a USER decision and is never derivable from
  `proGoAutoPublish` (test-enforced); the CLI writes the
  `[decision][review-accepted]` session-log entry itself.
- WEB-RUNBOOK §5: a review that finds nothing blocking publishes its feedback
  and then the `approval` immediately in the same Web turn (`feedback →
  approval` is legal; the feedback event is never skipped) — prevention at
  source; the CLI acceptance path covers sessions that end after the feedback.

## Feature B — Intent-alignment briefing gate (core)

- **Intent anchor**: `CONTRACT.json` gains `intents` (`INT-###` items in user
  language: statement + rationale); once declared, every REQ/INV/WF/NFR must
  carry non-empty `intentIds`, both directions fail-closed. Legacy contracts
  without intents stay legal. New designs MUST declare intents (WEB-RUNBOOK §4).
- **Alignment brief**: every Pro-authored `design`/`feedback` event pulled by
  `go`/`sync` requires `briefs/<event-id>/BRIEF.md` (user language) +
  `BRIEF.json` (machine-checkable) in the durable packet BEFORE any
  state-changing command (`report`, `accept-review`, `close` refuse with an
  actionable gate error). The brief covers every contract item / finding
  EXACTLY (no silent omission), classifying each as
  core/supporting/hardening/speculative/off-track with an intent path or an
  explicit no-intent-path marker plus a user-language purpose. Briefs propose;
  users decide: pending user decisions or a return-to-pro recommendation block
  until rulings are recorded with `decisions.confirmedBy: "user"`.
- **Legacy exemption**: gate arming is recorded per event in `STATE.json`
  `briefRequiredEventIds` (packet-state v2, additive) and only newly synced Pro
  events arm it — pre-feature in-flight flows are not retroactively bricked.
- **Decision feed-forward**: trim/defer rulings from briefs are injected into
  the next `webPrompt` as a binding "User scope rulings" section, so trimmed
  scope cannot silently re-inflate; Pro contests a ruling only via a new P0/P1
  finding declaring an intent path — which Feature A's eligibility gate then
  honors by forcing the Web roundtrip.
- New read-only `brief [flow]` subcommand prints the required roster, declared
  intents, expected paths, and a fill-in skeleton; `go`/`status` report an
  `alignmentBrief` status object.
- New generated artifact schema
  `pro-roundtrip-alignment-brief.schema.json` (NOT a protocol source).

## Downstream upgrade

After `vibe:sync`, the regenerated CONTRACT/EVENT-COMPLETE schemas and the
updated WEB-RUNBOOK/workflow-integrity re-derive a new content-addressed
protocol namespace automatically on the next `--publish` (v1.12.0 mechanics; no
manual protocol step). In-flight flows keep operating; only newly synced Pro
design/feedback events arm the briefing gate. Docs: SKILL.md "Alignment
briefing" / "Accepting a review from the CLI", CLI-CONTRACT command roster and
packet layout, workflow-integrity §12.
