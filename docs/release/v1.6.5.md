# v1.6.5

## Summary

Clarifies Codex Generator vs Codex Orchestrator maintenance modes and aligns `/vibe-sync` documentation with the current harness sync implementation.

## Changes

- Added an explicit Codex role-mode split to `AGENTS.md`: Sprint prompts run as Generator mode, while direct upstream harness maintenance can run as Codex Orchestrator maintenance mode.
- Updated orchestration docs to state that the Claude Orchestrator / Codex Generator table is the nominal harness mode, not a contradiction of direct Codex maintenance sessions.
- Clarified that Codex Orchestrator maintenance mode must use the provider-neutral `maintain-context` checkpoint workflow at work boundaries.
- Updated `/vibe-sync` skill instructions so post-sync verification uses harness-only typecheck behavior instead of product `tsconfig.json`.
- Removed stale guidance that missing `upstream` blocks sync or that exact `vX.Y.Z` refs are the standard manual setup. Exact refs are pins; caret refs float to compatible updates.
- Downgraded the `gap-rule-only-in-md` ledger entry from enforced coverage to pending enforcement: `vibe-rule-audit` surfaces uncovered rules but does not enforce every imperative rule by itself.
