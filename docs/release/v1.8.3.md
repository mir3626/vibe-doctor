# v1.8.3 - Verification Reuse and Faster Pro E2E

Date: 2026-07-20

## Changes

- Added an explicit harness verification manifest with seven root-test groups
  plus typecheck. Every root test has exactly one owner, known inputs select
  only their affected groups, and unknown harness changes fail closed.
- Added content-addressed, success-only verification receipts bound to group
  definitions, source and test contents, shared inputs, lockfiles, runtime,
  environment, and the verifier itself.
- Added cumulative Goal/Sprint verification from an exact base SHA. Manual
  verification and Stop QA use the same planner and receipts, while
  `vibe:self-test` and the release lane remain forced.
- Added a suite-wide Windows child-process preload so self-test descendants use
  hidden windows across spawn, exec, fork, npm, PowerShell, Git Bash, shared
  shell, and browser-open paths.
- Added an in-process `executeProRoundtrip()` boundary for Pro lifecycle
  behavior tests. The suite reuses a repository-owned bridge context but still
  performs real bare-remote commits, pushes, imports, reports, feedback,
  approval, close, stale-HEAD rejection, and tamper rejection.
- Replaced one `git log` process per immutable bridge payload with one
  flow-scoped history read. Every required file must still appear exactly once
  as an addition.

## Compatibility

- No Pro protocol, schema, flow archive, default-branch, or confirmation policy
  changed in this release.
- No root harness test was retired. The shipped `vibe-pro-go.mjs` wrapper smoke
  and the independent real-Git transport suite remain release boundaries.
- Prepared bridge-context injection is repository-bound and intended for
  serialized in-process execution. Normal CLI invocations continue to prepare
  and validate their own context.
- No migration is required. Downstream sync receives the new harness runtime,
  scripts, tests, skills, and documentation through existing manifest
  ownership.
