# v1.7.28 - Stop QA Successful-State Cache

Date: 2026-07-14

## Changes

- Added a SHA-256 fingerprint for Stop QA inputs: repository HEAD, changed code
  paths and contents, Node version/platform, and supported dependency lockfiles.
- Persisted only successful QA fingerprints in the ignored runtime path
  `.vibe/runs/stop-qa-cache.json`. The receipt is written through a temporary
  file and atomic rename.
- Made repeated Stop events on an unchanged successful state exit immediately.
  Hook-mode cache hits keep stdout and stderr empty; manual invocation prints a
  concise skip reason with the original QA log path.
- Kept failures uncached so they remain retryable. Relevant file contents,
  commits, runtime versions, or lockfiles automatically invalidate the cache.

## Root Cause

The Stop gate correctly skipped documentation-only work, but it treated every
dirty code tree as new work. A long-lived downstream worktree therefore ran the
entire project QA chain on every Stop even after that exact state had passed.
As product test suites grew, repeated turn completion expanded from roughly one
minute to several minutes without providing new verification evidence.

## Compatibility and Migration

- No data migration is required.
- Existing hook commands and failure JSON contracts are unchanged.
- The cache is disposable runtime state under an already ignored directory.
- The first Stop for a new code state still runs the configured QA chain; only
  a matching prior success is reused.
