# v1.6.1

## Summary

Hardens Windows, Git Bash, and WSL boundaries for Codex provider execution.

## Changes

- `src/lib/shell.ts` now treats explicit `.sh` commands on Windows as POSIX wrappers and runs them through Git Bash directly instead of relying on bare `bash` or `cmd.exe` path resolution.
- `commandExists()` now recognizes explicit script paths such as `./scripts/run-codex.sh`, including the Windows requirement that Git Bash must be available.
- `scripts/run-codex.sh` now fails early in WSL when `codex` resolves to a Windows npm shim under `/mnt/c` or `/c`.
- `scripts/run-codex.cmd --health` now calls npm `.cmd` shims correctly and returns normalized `codex-cli <version>` output.
- Added regression coverage for shell-script provider execution and native cmd wrapper health.
- Updated `/vibe-init`, README, provider, and Codex execution docs to define the standard OS boundary:
  - Codex provider config keeps `./scripts/run-codex.sh` on every OS.
  - Windows native execution uses Git Bash for `.sh` wrappers.
  - WSL uses Linux-installed `node` and `codex`, not Windows npm shims under `/mnt/c`.

## Migration Notes

- No state migration is required.
- Windows users should install Git for Windows or set `VIBE_GIT_BASH` to the Git Bash executable path if it is installed outside the default location.
- Existing local configs that directly call `codex.cmd exec` will continue to work, but the template standard is to keep the `run-codex.sh` wrapper so UTF-8 safety, retries, common rules, and session-start hooks remain active.
