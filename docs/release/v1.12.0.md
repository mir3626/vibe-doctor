# v1.12.0 - Content-Addressed Pro Protocol Namespaces

Date: 2026-07-23

Adopts the downstream drift handoff (P1): `vibe-sync` 1.10.1 → 1.11.0 changed four
of the five protocol source files without bumping the fixed `PROTOCOL_VERSION =
'v1'`, so every previously bootstrapped `vibe-pro-bridge` permanently failed —
`start` with `protocol hash/content mismatch` and `go` with `local common harness
hash differs from FLOW.json` — and the append-only protocol root left no in-band
recovery. The protocol version is now a **pure function of protocol content**, so
a harness release can never strand a bridge again: changed content simply
bootstraps a new namespace.

## Changes

- **Content-addressed version derivation**: `version = v1-<hash8>` where
  `<hash8>` is the first 8 hex of sha256 over the sorted `<name>:<sha256>` lines
  of the five CRLF-normalized protocol sources (COMMON-HARNESS.md,
  WEB-RUNBOOK.md, FLOW/CONTRACT/EVENT-COMPLETE schemas). `PROTOCOL.json` is
  never a derivation input (it embeds the version — self-reference). Namespace
  root: `protocol/v1-<hash8>/`. The fixed `PROTOCOL_VERSION`/`PROTOCOL_ROOT`
  exports are removed; `ProtocolBinding.version` is a string.
- **`ensureProtocol` per-namespace semantics**: computed namespace absent →
  clean bootstrap under the caller's existing `--publish` authorization (this is
  the automatic recovery path after any harness release); partial → fail-closed
  unchanged; all-present → CRLF-normalized content compare kept as
  defense-in-depth (truncated-hash collision, pre-created bad namespace). The
  compare tolerates checkout-time CRLF materialization (`core.autocrlf=true` is
  the Windows default and the orphan bridge branch carries no `.gitattributes`).
- **`verifyPinnedProtocol` generation gate**: a flow pinned to a different
  protocol generation now fails with an error naming BOTH versions (pinned vs
  local) and operator guidance. Active flows spanning a harness sync stay
  intentionally blocked; closed flows are never re-synced and keep their pinned
  namespace forever (append-only preserved).
- **Schema (additive)**: `flow.protocol.version` and event `protocolVersion`
  patterns widen to `^v[1-9][0-9]*(-[0-9a-f]{8})?$` — legacy `"v1"` documents
  on existing bridges still parse. JSON schemas regenerated from the Zod source.
- **CLI**: `webPrompt`, `doctor`, and the bootstrap hint interpolate the flow's
  actual pinned namespace instead of a hardcoded `protocol/v1`.
- **Docs**: `workflow-integrity.md`, `WEB-RUNBOOK.md` §3, and
  `bridge-runbook.md` switch to newest-namespace discovery ("select the
  `protocol/<version>/` whose files were added by the most recent commit").
- **Lineage rule**: the manifest's serialized shape (field order,
  `schemaVersion` literal, indentation) is NOT part of the hash — any change to
  `PROTOCOL.json` serialization requires a `PROTOCOL_LINEAGE` bump (`v1` →
  `v2`), documented at the constant.

## Downstream upgrade

After `vibe:sync`, the next `vibe:pro-go bootstrap`/`start` with `--publish`
bootstraps the new namespace automatically — no manual protocol surgery. Old
namespaces and closed flows are untouched. A flow that was ACTIVE across the
sync remains blocked by the generation gate: finish or close it with the harness
generation that created it, or start a new flow.
