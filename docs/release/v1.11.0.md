# v1.11.0 - Coordinated Cross-Flow Close

Date: 2026-07-22

Adopts the osint-stock-screener proposal "coordinated cross-flow close": when one
Pro approval decides several flows jointly, the CLI previously closed the primary
flow alone (the one action the joint-close invariant forbade) and left every
coordinated flow permanently open — its authorization existed only by reference
in another flow's approval, which `close` could neither read nor enforce. The
fix follows the pattern of `finalGatePolicy` and `productPlane`: **the approval
declares the coordination set; the close command enforces it atomically.**

## Changes

- **Approval-declared coordination set (schema, additive)**: an approval's
  `COMPLETE.json` may carry
  `coordinatedClose { jointInvariant: true, primaryFlowPath, flows[] }` where
  each member records its `flowPath` and `approvedBoundarySha`. The declaration
  is validated in isolation (`validateCoordinatedCloseDeclaration`): the
  approval must live in the declared primary flow, the primary must be a member
  at the approval's own frozen HEAD, and members must be unique. Absent the
  block, single-flow close is byte-for-byte unchanged.
- **By-reference member close (schema, additive)**: a `closed` marker may carry
  `authorizedByFlowPath` + `authorizedByEventId` (paired) referencing the
  primary approval, plus `coordinatedWith` listing the other members. Schema
  placement is enforced (`coordinatedClose` approval-only; authorization fields
  closed-only).
- **Chain carve-out**: `implementation-report`/`remediation-report` → `closed`
  becomes a legal transition ONLY when the closed marker carries a by-reference
  authorization; without it the transition fails exactly as before.
- **Pinned-commit cross-flow validation**: when a flow's snapshot contains a
  by-reference close, the loader reads the referenced approval from the SAME
  pinned bridge commit (never the worktree), and requires: the reference is a
  real approval, its declaration includes this flow as a member, and the close
  HEAD equals the declared `approvedBoundarySha`. Any mismatch fails the whole
  snapshot load.
- **Coordination-aware atomic `close`**: when the governing approval declares a
  set, `close <primary> --publish` resolves every member at one pinned commit
  and writes every remaining close marker in ONE append-only bridge commit —
  all or none. Already-closed members are accepted as satisfied (idempotent set
  completion), which is also the recovery path for a set whose primary was
  closed before this release. Fresh joint closes keep the full primary gates
  (HEAD currency, packet sync, matrix); residual completion closes members at
  their recorded approved boundaries without requiring repo HEAD currency,
  because the pinned approval — not repo state — is the authorization.
- **Refusal guard**: a close that would leave a declared set partially closed
  is refused: the coordinated path is the only way to close any member, and a
  non-primary member invocation is redirected to the primary flow.
- **`WEB-RUNBOOK.md` §6** documents authoring the declaration;
  `docs/context/workflow-integrity.md` §10 records the joint-close invariant.

## Compatibility

- All schema additions are optional; historical flows and markers keep
  re-validating unchanged, and flows whose approvals declare no coordination
  keep today's single-flow close exactly (the single-flow path was extracted,
  not modified).
- No new event kind: coordinated closes reuse `closed` with authorization
  fields, so event-id grammars, directory parsing, and rosters are untouched.
- The osint-stock-screener stuck state (primary `002` closed, coordinated
  `001` permanently open under the `2700` joint approval) becomes recoverable:
  after syncing v1.11.0, `close flows/20260720/002-quant-authority-trust-roots
  --publish` publishes the coordinated marker for flow `001` as an idempotent
  set completion — no bridge hand-editing.
