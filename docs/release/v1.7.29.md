# v1.7.29 - Nonblocking Harness-Only Stop QA

Date: 2026-07-14

## Changes

- Restricted automatic Stop QA detection to harness-owned and hybrid paths from
  `.vibe/sync-manifest.json`. Product-only `src/**`, `test/**`, and other
  project-owned changes no longer schedule Stop QA.
- Replaced the synchronous Stop QA process with a detached, hidden worker. The
  foreground hook only resolves the repository, fingerprints the relevant
  harness state, acquires an exclusive lease, schedules the worker, and exits
  successfully with empty stdout/stderr.
- Limited the worker to `npm run vibe:typecheck --silent` and
  `npm run vibe:self-test --silent`. It never invokes downstream product
  `vibe:qa`, `test`, `lint`, or `build` commands. The worker also removes the
  hook-only `CLAUDE_PROJECT_DIR` variable before running tests so nested hook
  fixtures resolve their own stdin cwd instead of the parent repository.
- Added an atomic versioned result receipt at
  `.vibe/runs/stop-harness-qa-state.json` and an exclusive worker lease at
  `.vibe/runs/stop-harness-qa-worker.lock`. Identical successful states and
  concurrent Stop events do not duplicate work; abandoned leases are reclaimed
  after a bounded stale interval.
- Persisted background failures with their exit status and log path. The next
  Stop reports that failure once through a valid nonblocking JSON
  `systemMessage`; later Stops stay silent until the relevant state changes.
- Kept direct CLI invocation synchronous for diagnosis. `--force` bypasses a
  successful cached state when an operator explicitly wants a fresh run.

## Root Cause

The v1.7.28 fingerprint removed repeated QA for an already successful dirty
state, but the first Stop for every new state still synchronously executed the
entire downstream project QA chain. In the affected project that chain took
roughly two and a half minutes, so Claude could not accept the next message.
It also reported two unrelated product tests that were actively being changed,
even though the Stop gate existed to protect harness maintenance.

## Compatibility and Migration

- No database or project-state migration is required.
- The obsolete v1.7.28 `stop-qa-cache.json` file is ignored and may be deleted;
  v1.7.29 uses a new schema-v2 result receipt.
- Existing Claude hook commands remain valid. Explicit `--hook` and stdin event
  auto-detection retain the cwd and JSON output contracts from v1.7.26/v1.7.27.
- A background failure is intentionally observed on the following Stop rather
  than blocking the Stop that scheduled it. The full log remains under
  `.vibe/runs/<date>/stop-harness-qa-*.log`.

## Verification

- Focused Stop gate tests for product-only exclusion, detached timing,
  concurrent deduplication, stale-lease recovery, success invalidation, manual
  exit propagation, and one-time background failure JSON
- `npm run vibe:typecheck`
- `npm run vibe:build`
- `npm run vibe:self-test`
- `npm run vibe:sync-audit`
