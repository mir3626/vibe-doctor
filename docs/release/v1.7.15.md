# v1.7.15 - Sprint-Mode Wildcard Permissions

Date: 2026-05-11

LTS: `v1.7.3-lts` remains the stable baseline. This is a mainline Claude permission-noise patch.

## Changes

- Updated `/vibe-sprint-mode` presets to current Claude Code wildcard permission syntax.
  - Replaced legacy `Bash(cmd:*)` rules with current wildcard command-family rules.
  - Extended mode now uses grouped allow rules for common Sprint command families such as `npm *`, `npx *`, `node *`, `git *`, `gh run *`, `gh api *`, read/file operation commands, and Windows `cmd /c` wrapper forms for the same families.
- Added preset-owned `permissions.deny` guards for critical operations:
  - force push
  - hard reset
  - git clean
  - destructive branch/delete operations
  - destructive checkout/restore/rebase in extended mode
  - `rm`, `rmdir`, `npm publish`, `gh pr` create/merge/close, and `gh release create`
  - nested Windows `cmd /c` critical forms for destructive file commands, force push, hard reset, git clean, branch deletion, restore/checkout/rebase, and `npm publish`
  - `.env*`, `secrets/**`, and `config/credentials.json` write/edit paths
- Extended `vibe-sprint-mode.mjs` so `on`, `off`, and `status` manage both allow rules and deny guards.
- `on` now removes old checked-in preset rules that used the legacy colon syntax before adding the new wildcard rules.
- Statusline now renders `sprint:off`, `sprint:core`, `sprint:extended`, or `sprint:partial` by comparing `.claude/settings.local.json` against the shipped presets.

## Why

Claude Code's current Bash permission syntax uses wildcard matching, not regex. The old preset contained `Bash(cmd:*)` entries that no longer matched real commands because `:` is treated literally. This caused repeated prompts even after `/vibe-sprint-mode extended`.

## Compatibility

- `/vibe-sprint-mode on --tier extended` updates ignored local `.claude/settings.local.json`; shared `.claude/settings.json` remains project-safe.
- `/vibe-sprint-mode off` removes both new wildcard preset rules and known legacy colon/broad `cmd /c *` preset rules while preserving custom user entries.
- Critical operations remain gated through deny rules even when extended mode uses broad command-family allow rules.
