# v1.1.1 — Sprint 완료 단일 커밋 원칙

**Date**: 2026-04-15
**Type**: patch (문서 전용, 코드 변경 없음)
**Version**: `harnessVersion` 1.1.0 → **1.1.1**

## Why

dogfood5 이후 사용자 피드백:

> `.vibe/agent/` 파일들은 context compact시 handoff를 위해 무조건 origin에 푸시되어야 하는 파일들이야? 게임으로 치면 일종의 클라우드 세이브파일 역할을 하는건가? 매번 커밋되다보니 실제 chore나 feature, refactor수정건같은 중요한 커밋들과 뒤섞여서 헷갈릴 수 있을 것 같아.

관찰:
- `.vibe/agent/{sprint-status.json, handoff.md, session-log.md}` 는 크로스-세션/머신/팀원 handoff를 위한 "클라우드 세이브" 역할이 맞다 → git push 필요
- 하지만 Sprint당 2 커밋(feature + `docs(sprint): close ...`) 구조는 `git log` 가독성을 해친다
- Sprint 30개 돌리면 state-only 커밋 30개가 feature 사이에 섞여 실제 변경 의미 추적이 어려움

## What

**Sprint 완료 시 단일 커밋 원칙**: Generator feature 파일 + state 3종을 **한 git commit** 에 묶는다. 별도 `docs(sprint): close ...` 커밋 금지.

`vibe-sprint-complete.mjs` 는 v1.0.x 부터 이미 자동 커밋을 하지 않고 파일 업데이트만 수행 — 정책 변경에 구조적으로 준비된 상태. 이번 패치는 Orchestrator 행동 규율의 **문서화**.

## Examples

**Bad (v1.1.0 이전):**
```
1948141 feat(sync): implement harness sync engine
425505e docs(sprint): close sprint-03-ui-skeleton → passed
6307145 feat(game): add whisper state machine
3909c70 docs(sprint): close sprint-02-whisper → passed
...
```

**Good (v1.1.1+):**
```
1948141 feat(sync): implement harness sync engine + sprint-03 close (LOC +420/-8)
6307145 feat(game): add whisper state machine + sprint-02 close (LOC +350/-12)
...
```

커밋 수 절반, 각 커밋이 "작업 + 상태 스냅샷" 단위로 의미 명확.
